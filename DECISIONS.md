# MathPlay 决策记录

> 这里记录所有"为什么选 A 不选 B"的产品/技术决策。
> 未来想推翻某个决策时,先读这里看当初的理由,避免循环讨论。

## 决策格式
每条决策包含:**问题** / **选择** / **放弃的备选** / **理由** / **可重新考虑的条件**

## 产品决策(8 条,2026-05-12)

### D1:子知识点范围
- **问题**:"时间和钟表"包含 6 个子概念,MVP 做哪些?
- **选择**:只做 #5(时针分针联动关系)
- **放弃**:#3 半点、#4 几点几分、#6 时间加减(留给 V2/V3)
- **理由**:#5 是纸笔教学最讲不清的,也是本产品**不可替代性**最强的点。MVP 做 1 个做透,胜过做 4 个做粗。
- **可重新考虑**:V2 时女儿玩完反馈"还想要更多"

### D2:形式组合
- **问题**:用游戏、动画、还是工具?
- **选择**:沙盒(可拖动钟表)+ 3 个挑战关卡
- **放弃**:纯故事动画(开发量大,重玩低);纯关卡(像题库,不爱玩);齿轮拆解(偏机械,抽象)
- **理由**:"Discovery before Drill" — 让她先自由探索发现规律,再用关卡巩固。这是 DragonBox/Khan Academy Kids 的成熟做法。
- **可重新考虑**:观察发现她对沙盒/关卡某一种特别冷淡

### D3:单次时长
- **问题**:目标体验时长?
- **选择**:约 10 分钟(沙盒 ~4 分钟 + 关卡 ~6 分钟)
- **放弃**:5 分钟(太短没效果);15+ 分钟(伤眼,MVP 不够)
- **理由**:对标"一次作业"体感,不伤眼,留有想念感
- **可重新考虑**:观察她真实的注意力曲线后

### D4:奖励机制
- **问题**:要不要积分/星星/解锁?
- **选择**:轻量反馈(动画+音效),无积分系统
- **放弃**:重度游戏化(星星/解锁/排行榜)
- **理由**:Deci & Ryan 自我决定理论 — 过度外在奖励摧毁内在动机。她的内在动机已存在,产品要做的是"帮她体会搞懂的快乐",不是"用糖引诱"。
- **可重新考虑**:不太可能改;这是教育哲学层面的选择

### D5:错误处理
- **问题**:答错怎么办?
- **选择**:分级提示(3 次后自动演示)
- **放弃**:直接告诉答案(治标);静默等待(8 岁会挫败)
- **理由**:错误是学习核心机会,引导式比告知式更有效
- **可重新考虑**:观察提示用语是否儿童化不够

### D6:进度记录
- **问题**:要不要保存她的游玩数据?
- **选择**:MVP 不做(给孩子的 + 给家长的都不做)
- **放弃**:localStorage 持久化进度;家长数据面板
- **理由**:MVP 只 3 关 10 分钟,关闭重玩零负担;最好的数据面板是"你坐她旁边看 5 分钟"
- **可重新考虑**:V3+ 有多个模块时,数据面板才有意义

### D7:陪玩模式
- **问题**:为独立玩还是陪玩设计?
- **选择**:主要独立玩,你偶尔陪
- **放弃**:必须陪玩(退化为辅导工具)
- **理由**:产品价值在于"她遇困难能反复独立操作直到搞懂",这是纸面做不到的
- **可重新考虑**:不太可能改

### D8:屏幕方向
- **问题**:横屏、竖屏、还是都支持?
- **选择**:只支持横屏,竖屏显示提示"请把 iPad 横过来 🔄"
- **放弃**:横竖屏都支持(工作量翻倍)
- **理由**:钟表是圆的,横屏两侧布局舒展;省一半工作量
- **可重新考虑**:V2+ 有用户强烈反馈时

## 技术决策(2026-05-12)

### T1:技术栈
- **选择**:纯 HTML + CSS + JavaScript,不用框架
- **放弃**:React/Vue(学习曲线陡);p5.js/Phaser(MVP 不需要游戏引擎)
- **理由**:vibe coding 新手友好,iPad 性能最好,AI 出 bug 时易看懂
- **可重新考虑**:某个特定模块(如 V3 复杂动画)可能用 p5.js

### T2:部署方式
- **选择**:GitHub Pages(公开仓库)
- **放弃**:本地文件传输(每次更新麻烦);Vercel/Netlify(对项目过重)
- **理由**:免费、自动、URL 永久;女儿在家任何 WiFi 可访问
- **可重新考虑**:几乎不可能

### T3:版本控制
- **选择**:`mathplay/` 作为独立 git 仓库
- **放弃**:用父目录已有的空 git 仓库(会和"知识库"职责混淆)
- **理由**:mathplay 是要独立部署的产品,职责清晰
- **可重新考虑**:将来想把所有项目放一个 monorepo 时

### T4:素材
- **选择**:免费开源 + emoji + AI 生成图
- **放弃**:自己画(慢);商业素材(贵);完全裸 UI(丑)
- **理由**:MVP 先验证玩法,素材后期可换
- **可重新考虑**:V2+ 时和女儿一起画原创素材

## 文案决策(2026-05-12,Step 4 后)

### C1:模式命名 — "沙盒" → "自由玩 / 自由模式"
- **问题**:UI 上的"沙盒"用词对 8 岁孩子是否友好?
- **选择**:
  - 模式名称(banner、提示文案):"**自由玩**"(3 字、活泼)
  - 返回按钮文案:"**返回自由模式**"(动作 + 目的地,语义完整)
  - 代码内部:保留 `sandbox`(class/变量名)
- **放弃**:"沙盒模式"(工程术语,孩子不懂)、"随便玩"(略随意)、统一"自由玩"按钮(动词缺失)
- **理由**:孩子能秒懂;"返回..."比"回..."更书面化、按钮上看着更像一个"功能动作";内部代码保持英文 `sandbox` 是合理工程实践(命名空间清晰,改文案不动 JS)
- **尺寸适配**:按钮 padding 用 em 单位自动伸缩;加 `white-space: nowrap` 防止小屏中文换行
- **可重新考虑**:女儿试玩反馈不喜欢

### C2:关卡提示语
- **问题**:挑战 banner 的提示应该怎么写?
- **选择**:"**请拨动分针,使时间指向 X:XX**"
- **放弃**:"把分针拨到 X:XX"(语义不准,X:XX 是时间不是分针位置)
- **理由**:精确表达"分针是操作对象、时间是目标结果",符合教学逻辑(因→果)
- **可重新考虑**:孩子反馈太长读不完
